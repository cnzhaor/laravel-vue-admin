# 项目开发规则

## 适用范围

- 本规则适用于整个仓库。
- `backend/` 或 `frontend/` 下如果以后存在更具体的 `AGENTS.md` 或 `AGENTS.override.md`，以更接近目标文件的规则为准。
- 任务中的明确要求优先于本文件中的默认约定。

## 沟通与执行

- 使用中文沟通、说明变更并编写项目文档。
- 修改前先阅读相关代码、配置和测试，确认现有实现及影响范围。
- 只修改完成当前任务所必需的内容，不顺带重构无关代码。
- 保留用户已有修改；发现工作区存在无关改动时，不覆盖、不回退、不纳入当前任务。
- 需求存在轻微歧义时，采用风险最低且容易回退的实现，并在交付说明中注明假设。
- 涉及数据丢失、生产环境、外部发布或不可逆操作时，必须先获得用户确认。

## 技术栈与架构

- 保持 Laravel 13、PHP 8.3、Vue 3、TypeScript、Vite、Pinia、Vue Router 和 Element Plus 技术栈。
- 项目保持前后端分离，通过 Docker Compose 运行和验证。
- 后端 API 继续使用 `/api/v1` 版本前缀。
- 认证继续使用 Laravel Sanctum Cookie Session 和 CSRF 防护。
- 数据库存储 UTC 时间，前端按照浏览器本地时区显示。
- 未经用户明确同意，不新增或替换生产依赖，不更换核心框架和基础设施。

## 后端规范

- 控制器负责接收请求、调用业务逻辑和返回响应；复杂业务逻辑放入 Service。
- 所有外部输入必须经过明确验证，复杂验证优先使用 Form Request。
- 数据查询应考虑权限、数据范围、分页、排序、空值和异常情况。
- 涉及数据修改的功能应检查权限，并按现有机制记录操作日志。
- 数据库结构变化必须通过 migration 实现，不直接修改生产数据库结构。
- 初始化数据变化同步维护 Seeder；测试数据优先使用 Factory。
- 密钥、Token、密码等敏感信息只能从环境变量或安全配置读取，不得写入代码、日志或响应。
- 新增或修改业务行为时，补充对应的 Feature 或 Unit 测试。

## 前端规范

- Vue 组件使用 `<script setup lang="ts">` 和 Composition API。
- 保持严格的 TypeScript 类型；没有充分理由不得使用 `any`。
- API 请求统一放在 `frontend/src/api/`，共享类型放在 `frontend/src/types/`。
- 可复用逻辑和 UI 应放入适当的共享组件或组合式函数，避免页面间复制。
- 页面权限、路由权限和按钮权限必须与后端权限保持一致。
- 用户可见的提示和错误信息使用清晰、友好的中文。
- 日期时间统一通过现有格式化方式转换为浏览器本地时区。
- 修改页面或交互时，检查加载、空数据、失败、禁用和重复提交状态。

## API 与数据库兼容性

- 修改已有 API 的路径、请求字段、响应结构或错误语义前，评估前后端兼容性。
- 能通过新增可选字段实现的，不无故进行破坏性变更。
- 前后端涉及同一业务字段时必须同步修改类型、校验、接口调用和显示逻辑。
- migration 应考虑已有数据；高风险数据迁移必须先说明影响和回滚方案。

## 设计与独立审查

- 实现前先写出最小行为契约：用户术语、API 字段、UI 文案、状态转换和实际执行语义必须一致；同一概念出现不同命名或含义时，先澄清再编码。
- 自行实现调度、重试、锁、限流、缓存、事务或异步机制前，必须检查 Laravel、Vue 及当前依赖是否已提供原生能力；能直接满足需求时优先使用原生机制。
- 禁止用长时间阻塞、轮询或占用 Worker 的方式模拟框架已支持的延迟、调度或重试行为；如确需模拟处理耗时，字段、文案和测试必须明确表达“处理耗时”，不得冒充“延迟执行”。
- 实现和自动化验证完成后，必须单独执行一轮设计 Review；测试、构建、路由检查和端到端演示仅证明实现可运行，不能代替 Review。
- Review 必须重新对照原始需求和完整 diff，不以现有实现为前提；至少检查语义一致性、框架原生能力、资源占用、并发、幂等、事务边界、超时、重试、失败路径、权限和可观测性。
- 测试应优先断言业务不变量和外部可观测结果，不得只复制当前实现细节；延迟、队列、事务和并发功能必须覆盖等待期、执行期和终态。
- 队列、并发、事务、认证权限、支付返利或外部 API 等高风险变更，在可用时应将原始需求与 diff 交给独立审查者或子代理复核，不提供实现者的预设结论；无法独立复核时，必须在交付中明确说明。
- Review 发现命名、UI、API、实现或测试之间存在语义不一致时，必须先修正并重新验证，不得以“演示可用”或“测试通过”作为交付理由。

## 代码与文档同步

代码改动完成时，必须检查并更新受影响的文档：

- 功能模块、安装方式、访问地址、常用命令或项目结构变化：更新根目录 `README.md`。
- 环境变量新增、删除或语义变化：同步更新相应的 `.env.example` 和 `README.md`；示例不得包含真实密钥。
- API 路由、参数、响应、权限或错误码变化：更新对应 API 文档；当前没有独立 API 文档时，在根目录 `README.md` 中补充对应模块说明。
- 数据库表、字段、索引或初始化数据变化：同步维护 migration、Seeder、测试以及相关数据说明。
- Docker、Nginx、端口、服务或部署流程变化：更新 `README.md` 的部署、配置和常用命令章节。
- 新增、删除或调整用户可见功能：更新 `README.md` 的功能模块说明。
- 当前运行状态、已完成事项、验证结果、已知问题或待办发生变化：更新 `HANDOFF.md`。
- 修复故障且该故障可能再次发生：在 `README.md` 的常见问题中记录现象、原因和处理方法。
- 仅内部重构且外部行为、配置和操作方式完全不变时，可以不更新用户文档，但交付时必须说明无需更新的判断。
- 禁止只修改代码而遗漏必要文档；交付结果必须单独列出已更新的文档或说明为什么无需更新。

## 验证要求

- 后端代码修改后运行：`docker compose exec app php artisan test`。
- 后端路由修改后运行：`docker compose exec app php artisan route:list`，必要时使用 `--path` 缩小范围。
- 前端代码修改后运行：`docker compose exec frontend npm run build`。
- UI、路由或交互修改后，应在浏览器中实际检查关键流程。
- migration 或 Seeder 修改后，应在可控测试环境验证，禁止为验证而清空用户现有数据库。
- 修改配置或部署文件后，至少检查配置语法和容器状态。
- 验证失败时继续定位属于本次改动的问题；如果是既有问题，明确区分并提供证据。
- 无法执行某项验证时，必须说明未验证内容和原因，不得声称已经通过。

## 安全规则

- 不提交真实 `.env`、密码、Access Token、AppSecret、Cookie、数据库备份或个人数据。
- 不在日志、测试快照、截图和交付说明中泄露敏感信息。
- 生产环境必须关闭 `APP_DEBUG`，并使用独立强密码和最小权限账号。
- 权限、认证、支付、返利和外部平台接口属于高风险区域，修改时必须补充失败路径和越权检查。
- Adminer 等管理工具不得直接暴露到公网。

## Git 规则

- 修改前后检查 `git status` 和相关 diff，避免混入无关改动。
- 不回退、覆盖或删除用户已有变更。
- 一个提交只包含一个清晰目的，提交信息优先使用 Conventional Commits，例如 `feat:`、`fix:`、`docs:`、`test:`、`refactor:`。
- 未经明确要求，不执行 commit、push、merge、rebase、创建分支或创建 PR。
- 提交前确认没有密钥、缓存、依赖目录、构建产物或临时文件被误纳入版本控制。

## 完成交付

完成任务时简洁说明：

- 实现或修复了什么；
- 修改了哪些主要文件；
- 更新了哪些文档，或为何无需更新；
- 执行了哪些验证及结果；
- 设计 Review 的结论、发现并修正的问题，以及是否完成独立复核；
- 仍存在哪些风险、限制或待办。
